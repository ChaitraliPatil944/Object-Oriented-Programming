# Lesson 162 — Bytecode

> **Object-Oriented Programming (OOP) Handbook**

# 1. Introduction

Java is often described as **"Write Once, Run Anywhere (WORA)"**. The main reason this is possible is **Bytecode**.

Instead of compiling Java source code directly into machine code, the Java compiler converts it into an intermediate language called **bytecode**, which is executed by the Java Virtual Machine (JVM).

Understanding bytecode is essential for JVM and OOP interviews because every Java program ultimately runs as bytecode.

---

# 2. What is Bytecode?

**Bytecode** is the platform-independent instruction set generated by the Java compiler (`javac`).

It is stored in `.class` files and is executed by the JVM.

```text
Student.java
      │
    javac
      ▼
Student.class
(Bytecode)
      │
      ▼
JVM
      │
Machine Code
```

---

# 3. Compilation Process

```text
Java Source Code
        │
      javac
        │
        ▼
Bytecode (.class)
        │
        ▼
Java Virtual Machine
        │
        ▼
Machine Instructions
```

The JVM translates bytecode into machine code for the host operating system.

---

# 4. Why Bytecode?

Advantages:

- Platform independence
- Better security
- Smaller executable files
- JVM optimizations
- JIT compilation support

---

# 5. Viewing Bytecode

Java provides the **javap** tool.

Compile:

```bash
javac Student.java
```

View bytecode:

```bash
javap -c Student
```

The `-c` option displays bytecode instructions.

---

# 6. Example

Java source:

```java
public class Demo {

    public int add(int a, int b) {
        return a + b;
    }
}
```

Simplified bytecode:

```text
iload_1
iload_2
iadd
ireturn
```

Each instruction is executed by the JVM.

---

# 7. Common Bytecode Instructions

| Instruction | Purpose |
|-------------|---------|
| `iload` | Load integer |
| `istore` | Store integer |
| `iadd` | Add integers |
| `isub` | Subtract integers |
| `imul` | Multiply integers |
| `idiv` | Divide integers |
| `invokevirtual` | Call instance method |
| `invokestatic` | Call static method |
| `return` | Return from method |

---

# 8. Bytecode Execution

```text
.class File
     │
     ▼
Bytecode Verifier
     │
     ▼
Interpreter
     │
     ▼
JIT Compiler
     │
     ▼
Machine Code
```

The JVM first verifies the bytecode, then interprets or compiles it.

---

# 9. Bytecode Verification

Before execution, the JVM checks that bytecode:

- Is valid
- Does not corrupt memory
- Does not violate access rules
- Follows JVM specifications

This contributes to Java's security model.

---

# 10. Bytecode and OOP

OOP concepts are represented in bytecode:

- Object creation
- Method calls
- Inheritance
- Interfaces
- Polymorphism
- Exception handling

For example, object creation uses instructions such as:

```text
new
dup
invokespecial
```

---

# 11. Best Practices

- Understand that `.class` files contain bytecode, not machine code.
- Use `javap` to inspect compiled classes.
- Remember that bytecode is platform-independent.
- Know that the JVM translates bytecode into native instructions.

Because every elegant Java class eventually becomes a stream of tiny instructions. The JVM is basically an extremely patient translator.

# 15. Summary

Bytecode is the intermediate, platform-independent instruction set generated by the Java compiler and stored in `.class` files. The JVM verifies and executes this bytecode using an interpreter and the Just-In-Time (JIT) compiler, enabling Java's "Write Once, Run Anywhere" capability while supporting efficient execution and strong security.
